# EasyWords 项目讨论经验

> 记录开发过程中的思想碰撞、问题分析和结论。长期维护。

---

## 2026-03-20 自由对话模式语音丢失问题（完整修复过程）

### 问题现象
用户在自由对话模式下说话，系统提前发送了不完整的语音内容。例如：
```
San Francisco. Where is it? Can you tell me which is? Which part of San Francisco are you in? And.
```
用户实际上还在继续说话，但系统只记录到 "And" 就发送了。

---

### 第一轮分析：识别器重启清空缓存

**错误方向**：AI 认为问题是静音检测时间太短（2秒），建议延长。

**用户反馈**：
> "这个是在增加时长，对体验不好的。系统已经记录了最后的 'And'，我后面没有停顿就继续说了，但系统直接发送了。"

**根因**：识别器重启时 `onstart` 和 `start()` 会清空 `finalTranscript`。

**修复**：
- `onstart` 不再清空 transcript
- `start(clearPrevious)` 添加参数控制是否清空

**结果**：问题仍存在。

---

### 第二轮分析：重复发送

**日志发现**：
```
[FreeTalk] stop() 返回文本: Hello, can you hear me? And uh.
[FreeTalk] 发送文本: ...                           ← 第一次发送
[FreeTalk] 静音2秒，开始发送倒计时                   ← 倒计时没被取消
[FreeTalk] 发送倒计时结束                           ← 第二次触发
```

**根因**：发送后 `interimTextRef.current` 没清空，倒计时定时器没取消。

**修复**：`sendFreeTalkMessage` 中立即清空缓存并取消所有定时器。

**结果**：问题仍存在。

---

### 第三轮分析：识别器停止导致发送流程被卡

**日志发现**：
```
[ASR] onend 触发
[FreeTalk] 识别器意外停止，立即重启
[ASR] onstart 触发
[FreeTalk] 识别器仍在工作，延迟 500ms 后重试  ← 无限循环
```

**根因**：识别器重启后 `isListening()` 返回 true，导致 `doSendFreeTalkMessage` 无限延迟。

**修复**：添加 `isSendingRef` 标志，发送时不重启识别器。

**结果**：问题仍存在。

---

### 第四轮分析：发送倒计时没被正确取消

**用户反馈**：
> "系统已经录到了最后的读音，按照逻辑，只要录到了读音，就要重新走等待逻辑。"

**日志发现**：
```
[FreeTalk] 发送倒计时结束，发送消息
[FreeTalk] doSendFreeTalkMessage 被调用
[ASR] onresult 触发, text: but        ← 新内容在发送后才到达
```

**根因**：`stop()` 调用后，识别器还在返回新结果，但发送流程已经开始了。`onresult` 没有取消发送。

**最终修复**：
1. `onresult` 中检查 `isSendingRef`，如果有新内容就取消发送
2. `doSendFreeTalkMessage` 中 `stop()` 返回后再次检查是否被取消

**结果**：✅ 问题解决

---

### 核心经验

1. **不要只看表面现象**：静音检测时长看起来是问题，实际不是
2. **日志是关键**：通过详细日志才能发现真正的时序问题
3. **异步流程的状态管理**：发送是一个异步流程，任何一步都可能被新事件打断，需要设计取消机制
4. **用户反馈很重要**：用户说"系统录到了最后读音"是关键线索，说明问题不在识别，而在发送逻辑
